## 漏洞报告流程

### 报告渠道

我们重视安全研究者的贡献，欢迎负责任地披露漏洞。

**主要渠道**:
- 📧 邮箱: **security@cis.dev**
- 🔐 PGP 公钥: [security.asc](https://cis.dev/security.asc) (指纹: `A1B2 C3D4 E5F6 7890`)

**备用渠道** (如果主要渠道不可用):
- GitHub Security Advisories (仅限高危漏洞)

### 报告内容

请尽可能包含以下信息：

1. **漏洞描述**: 清晰的问题描述
2. **影响版本**: 受影响的 CIS 版本
3. **复现步骤**: 详细的复现方法
4. **影响评估**: 潜在影响和攻击场景
5. **缓解建议**: 如有可能，提供修复建议
6. **概念验证**: 可选，如有 PoC 代码

### 响应流程

```
收到报告 (0h)
    │
    ▼
初步确认 (24h) ──► 确认收到报告，分配跟踪编号
    │
    ▼
严重性评估 (72h) ──► 评估影响，确定严重等级
    │
    ├─► P0 (紧急): 立即开始修复
    ├─► P1 (高危): 7 天内修复
    ├─► P2 (中危): 下次发布修复
    └─► P3 (低危): 按需修复
    │
    ▼
修复开发 (时间依严重度而定)
    │
    ▼
安全更新发布
    │
    ▼
公开披露 (通常在修复后 30 天)
```

### 沟通承诺

- **24 小时内**: 确认收到报告
- **72 小时内**: 初步评估和严重等级确认
- **修复期间**: 每周进度更新
- **公开披露**: 事先通知报告者

### 奖励计划

我们提供漏洞奖励计划 (VRP)：

| 严重等级 | 奖励范围 |
|----------|----------|
| P0 - 关键 | $2,000 - $5,000 |
| P1 - 高危 | $500 - $2,000 |
| P2 - 中危 | $100 - $500 |
| P3 - 低危 | 感谢信 + 荣誉榜 |

**奖励条件**:
- 首次报告
- 负责任披露
- 提供详细报告和复现方法

## 安全更新政策

### 更新类型

| 类型 | 说明 | 发布频率 |
|------|------|----------|
| **安全补丁** | 修复安全漏洞 | 按需发布 |
| **维护更新** | 依赖更新、小修复 | 每月 |
| **功能发布** | 新功能、重大改进 | 每季度 |

### 安全通知

通过以下渠道发布安全通知：

1. **邮件列表**: security-announce@cis.dev
2. **GitHub**: Security Advisories 页面
3. **Discord**: #security-announcements 频道
4. **Twitter/X**: @CIS_Security

### 更新建议

**紧急更新** (P0/P1 漏洞):
- 建议 24-48 小时内应用
- 可能包含破坏性变更

**常规安全更新** (P2/P3):
- 建议 1 周内应用
- 向后兼容

### 签名验证

所有发布文件均使用 GPG 签名：

```bash
# 验证发布文件签名
gpg --verify cis-1.0.0.tar.gz.asc cis-1.0.0.tar.gz

# 导入 CIS 发布公钥
gpg --keyserver keyserver.ubuntu.com --recv-keys A1B2C3D4E5F67890
```

## 安全功能概览

CIS 内置以下安全机制：

| 层级 | 机制 | 说明 |
|------|------|------|
| **身份** | DID + Ed25519 | 去中心化身份，硬件绑定 |
| **传输** | Noise Protocol | 端到端加密通信 |
| **存储** | ChaCha20-Poly1305 | 数据库和私钥加密 |
| **访问** | ACL + 白名单 | 细粒度访问控制 |
| **执行** | WASM 沙箱 | Skill 隔离执行 |
